# ADR-006: Multi-Module Terraform with Bash Orchestration

**Date:** 2026-06-04
**Status:** Accepted
**Context:** Week 6 — EKS Validation

## Context

Week 6 deploys Atlas onto AWS EKS. This requires Terraform code for:

- VPC (network layer)
- EKS cluster (Kubernetes layer)
- IAM roles for service accounts (IRSA, workload identity)

Three layers with clear dependencies: EKS depends on VPC, IRSA depends on
EKS's OIDC provider. The question is how to structure these in Terraform.

The constraints driving the decision:

1. **Lifecycle is short.** Apply → demo → destroy in a single weekend session.
   No long-running cluster, no team collaboration on state.
2. **Cost discipline.** Each module's apply costs real money. Want ability to
   apply pieces independently for debugging.
3. **Local-first debugging.** Want each module to be plan-able in isolation
   without needing to apply prerequisites.
4. **Solo developer.** No team workflow, no concurrent edits to manage.

## Options Considered

### Option 1: Single root Terraform configuration

Put all resources (VPC + EKS + IRSA) into one root `main.tf` (or call them
as modules from a top-level config). Single `terraform apply` creates everything.

**Pros:**
- Outputs flow automatically via Terraform's dependency graph
- One state file
- Standard Terraform pattern for monolithic infrastructure

**Cons:**
- Cannot apply VPC alone for testing
- Single state file = whole-or-nothing for state ops (lock, refresh, destroy)
- A single resource failure aborts everything — partial state recovery is harder
- The blast radius of a typo touches all layers

### Option 2: Terragrunt

Use Terragrunt for DRY configuration and to define module dependencies
declaratively. Run `terragrunt run-all apply` to apply everything in order.

**Pros:**
- Battle-tested for multi-module orchestration
- Dependencies declared in code (`dependency` blocks)
- DRY backend configuration
- Industry standard for serious Terraform projects

**Cons:**
- Adds a new tool to the project (binary, install, learning curve)
- Atlas doesn't have the complexity Terragrunt was designed for
  (multiple environments × multiple regions × DRY-by-need)
- Documentation overhead — every reader now needs Terragrunt context
- One more thing that can be misconfigured

### Option 3: Remote state with S3 backend + cross-module data sources

Each module is independent, but they reference each other via
`data "terraform_remote_state"` reading from S3.

**Pros:**
- Standard production pattern
- Modules truly independent — separate state per module
- Locks via DynamoDB

**Cons:**
- Requires S3 bucket + DynamoDB table to exist BEFORE first apply
  (bootstrapping problem)
- More AWS resources to manage for a short-lived project
- Bucket + table can themselves be created via Terraform but require
  manual setup OR another bootstrap layer
- Atlas's "destroy everything when done" pattern means tearing down the
  backend layer too — adds complexity

### Option 4 (Chosen): Separate module directories + bash orchestration

Three directories: `vpc/`, `eks/`, `iam-irsa/`. Each has its own
`terraform.tfstate` locally. A `bootstrap.sh` script applies them in order,
pipes outputs from one module's `terraform output -json` into the next
module's `terraform.tfvars` via `jq`.

**Pros:**
- Each module remains plan-able and apply-able independently
- Bash makes the orchestration transparent — anyone reading the script
  sees exactly what happens
- No additional state infrastructure (no S3 bucket, no DynamoDB)
- "Maximum debug local" principle — every layer can be syntax-checked
  and plan-validated against AWS before any apply
- Symmetrical: `bootstrap.sh` to apply, `destroy.sh` to tear down

**Cons:**
- Bash + JSON manipulation is more fragile than pure Terraform
  dependency graphs (mitigated by using `jq` not regex)
- Three state files to track — each module's state is local to its directory
- Doesn't scale to team collaboration (not a goal for Atlas)
- Less "standard" — interviewers expecting Terragrunt might initially ask

## Decision

Use **separate Terraform module directories with bash orchestration**.

Layout:

infrastructure/terraform/
├── vpc/             # VPC, subnets, NAT, route tables
├── eks/             # Control plane, node group, OIDC
├── iam-irsa/        # IRSA roles for workloads
├── bootstrap.sh     # Apply all three in dependency order
├── destroy.sh       # Reverse order teardown
└── README.md

The scripts use `terraform output -json` and `jq` to pipe values
between modules. Each module's `terraform.tfvars` is regenerated by
the bootstrap script before that module is applied.

## Consequences

### Positive

- **Each Terraform layer is independently understandable.** Reading
  `vpc/main.tf` requires no knowledge of EKS or IRSA.
- **Cost recovery is granular.** If EKS apply fails halfway, VPC is still
  there. Re-running bootstrap.sh resumes cleanly.
- **The bash bootstrap is debugged code, not magic.** Every step is
  visible; nothing hidden behind a tool's DSL.
- **Cleanup is a single command.** `destroy.sh` reverses the apply order
  and ends with $0 AWS resources running.

### Negative

- **Bash + JSON manipulation has edge cases.** A malformed Terraform
  output, a missing field, or a network error during `jq` could leave
  the modules in inconsistent state. Mitigated by `set -euo pipefail` and
  explicit error checking after every `terraform output` call.
- **State files live in each module directory.** They are gitignored,
  but losing a state file (e.g., system rebuild) means we can't manage
  those resources via Terraform anymore. For Atlas's short lifecycle,
  this is acceptable; for production, S3 remote state would be required.
- **No locking.** Two concurrent `terraform apply` calls in the same
  module would corrupt state. Not a real risk for solo development;
  would be unacceptable in a team.

### Neutral

- **Three lock files committed to git** (`.terraform.lock.hcl` per module).
  This is correct Terraform practice — lock files pin provider versions
  and should be committed.
- **The bash script becomes a portfolio artifact.** A 250-line orchestration
  script demonstrates real engineering across shell + Terraform + JSON,
  which is more interview-worthy than a single `terragrunt run-all apply`.

## Validation

The approach was validated through:

1. Each module's `terraform plan` succeeded independently with correct
   resource counts (VPC: 18, EKS: 9, IRSA: 6 — total 33).
2. `bash -n` syntax validation of bootstrap.sh and destroy.sh.
3. `shellcheck` on both scripts (zero warnings).
4. Phase B will validate end-to-end apply + destroy.
