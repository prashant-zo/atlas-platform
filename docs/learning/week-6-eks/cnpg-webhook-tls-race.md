# CNPG Webhook TLS Race On Fresh EKS Deploy

**Date:** 2026-06-07
**Context:** Week 6 Phase B Take 2, fresh EKS bootstrap
**Status:** Resolved, pattern documented for future deployments

---

## What Happened

During Atlas's fresh EKS deployment, ArgoCD synced the platform layer successfully. All 7 platform Applications (argo-rollouts, cnpg-operator, ingress-nginx, kube-prometheus, loki, metrics-server, root-app-of-apps) reached Synced + Healthy.

But the workload `three-tier-dev` Application failed sync with:

Internal error occurred: failed calling webhook "mcluster.cnpg.io":
tls: failed to verify certificate: x509: certificate signed by unknown authority

Three CNPG resources (`Cluster`, `Pooler`, `ScheduledBackup`) couldn't be created.
After 5 retries, ArgoCD marked the operation Failed and stopped retrying.

## Root Cause

CNPG (CloudNative-PG) operator uses an admission webhook to validate Cluster CRDs at creation time. The webhook requires TLS, with a self-signed CA generated by the operator on first startup.

On a fresh deploy:

1. ArgoCD applies platform-cnpg-operator → operator pod starts (T+0)
2. Operator generates self-signed CA + cert, mounts to webhook serving secret
3. Operator injects CA bundle into `mcluster.cnpg.io` MutatingWebhookConfiguration
4. API server starts trusting webhook (T+30s typically)

Meanwhile, ArgoCD's wave-2 sync of three-tier-dev tries to create a CNPG Cluster CRD (T+15s, before webhook trust is established).

The API server forwards the admission request to the webhook → TLS handshake fails because API server doesn't yet trust the operator's CA → CNPG Cluster creation rejected.

ArgoCD retries 5 times with exponential backoff. Total retry window: ~3 minutes. If webhook cert propagation takes longer (e.g., during EKS resource contention), the retry limit is exhausted before the webhook becomes trustable.

After retries exhausted, ArgoCD marks operation as `Failed` and does NOT auto-retry on subsequent sync cycles. The Application stays "OutOfSync" indefinitely until manually re-synced.

## Resolution

After the webhook cert propagated (verified by curl test from inside cluster), a manual sync trigger created all CNPG resources successfully:

```bash
kubectl patch application three-tier-dev -n argocd \
  --type merge \
  -p '{"operation":{"sync":{"revision":"HEAD","syncOptions":["CreateNamespace=true","ServerSideApply=true"]}}}'
```

Within 30 seconds:
- CNPG Cluster `pg` created
- Pooler `pg-pooler-rw` created
- ScheduledBackup `pg-daily` created

Within 3 minutes:
- pg-1 primary instance Ready
- pg-2, pg-3 replicas Ready, streaming from pg-1
- Backend init containers completed (nc -z pg-pooler-rw 5432 check passed)
- Backend pods Ready 1/1

## Why ArgoCD Doesn't Auto-Recover

ArgoCD's `Retry Limit: 5` is the default for sync policies. After exhaustion, the failed operation is preserved in `status.operationState`. ArgoCD's self-heal mode triggers new syncs when it detects drift, but a stuck `operationState.phase: Failed` blocks new operation initiation.

The `refresh: hard` annotation only updates ArgoCD's comparison cache, not the operation state. To force a new sync, an explicit `operation` patch is required.

## CNCF-Aligned Fixes

This is a well-known bootstrap-order issue with operator-managed admission webhooks. Standard remediations:

1. **Sync waves** — ArgoCD sync waves allow specifying that wave-1 (operators) must complete before wave-2 (resources). With proper waves, the CNPG operator's webhook setup completes before any CNPG Cluster CRDs are applied.

2. **Increased retry limit** — Bump `retry.limit` from 5 to ~20 with `retry.backoff.maxDuration: 5m`. This gives the webhook more time to become available before ArgoCD gives up.

3. **PreSync hooks** — Add a pre-sync hook that verifies the webhook is responding before applying CNPG resources.

For Atlas, the recommended fix is **sync waves**:
- Wave -1: Operators (cnpg-operator, argo-rollouts, etc.)
- Wave 0: Workload (three-tier-dev)

## Verification Commands

When this issue recurs on future deploys, use these to diagnose:

```bash
# Is the webhook cert provisioned?
kubectl get secret -n cnpg-system cnpg-webhook-cert

# Does the CA bundle exist in webhook configurations?
kubectl get mutatingwebhookconfiguration cnpg-mutating-webhook-configuration \
  -o jsonpath='{.webhooks[0].clientConfig.caBundle}' | head -c 50

# Test webhook connectivity from inside cluster
kubectl run debug --rm -it --image=curlimages/curl --restart=Never -- \
  curl -sk https://cnpg-webhook-service.cnpg-system.svc:443/mutate-postgresql-cnpg-io-v1-cluster

# Expected response: HTTP 400 with content-type error (proves TLS handshake succeeded)
```

If the curl test returns a TLS error, the webhook isn't ready. Wait 30-60 seconds and retry.

If TLS handshake succeeds (returns the 400 JSON), trigger a fresh sync via the kubectl patch above.

## Lessons For Atlas

1. **Operator webhook bootstrap takes 30-90 seconds** — not instant.
2. **ArgoCD's default retry limit (5) is insufficient** for operator-dependent resources.
3. **Manual sync recovery is required when retries exhaust** — `refresh: hard` alone doesn't trigger new operations.
4. **Cross-check ArgoCD status with `kubectl get <resource>`** — ArgoCD's operationState can be stale; cluster ground truth is authoritative.

This pattern will recur in Phase B+1 unless Atlas implements proper sync waves. Documented as Phase B Take 2 known issue with mitigation steps above.
